## Why Starting Files Matter

A ChatGPT Project without starting files is a session waiting to drift. When scope isn't declared, the assistant fills it in from context. When constraints aren't written, they get ignored. The session may produce output — but it will be output shaped by assumptions, not by your actual intent.

<Note>
  A vague project start produces vague work. Project Forge makes the start concrete.
</Note>

Starting files don't constrain your work — they anchor it. They give both you and the assistant a shared reference point that can be consulted, updated, and passed forward into future sessions without losing continuity.

## How to Use Project Forge

<Steps>
  <Step title="Open Project Forge on GitHub and read the template structure">
    Review the four sections — scope, sources, constraints, and starting basis — before filling anything in. Understand what each section is asking for.
  </Step>
  <Step title="Fill in the four starting sections">
    Work through scope, sources, constraints, and starting basis for your specific project. Be concrete. Vague entries here produce the same problems as no entries at all.
  </Step>
  <Step title="Create a ChatGPT Project and upload the completed files">
    Upload your filled-in starting files to the ChatGPT Project as project knowledge. They become the persistent foundation the assistant will reference.
  </Step>
  <Step title="Reference the starting files explicitly in your first session message">
    Don't assume the assistant will read the files unprompted. Open each session by pointing to the starting files and confirming the assistant has oriented itself against them.
  </Step>
  <Step title="Update the starting basis at the end of each major session">
    After significant work, revise the starting basis to reflect the current state. This keeps future sessions accurate and prevents stale context from compounding.
  </Step>
</Steps>
